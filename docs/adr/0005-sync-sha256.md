# ADR 0005: Incremental Sync Engine with SHA256 Verification

- Status: Accepted (Supersedes ADR-0002)
- Date: 2026-03-16
- Author: Luis / Legacy Audio Project
- Supersedes: ADR-0002

## 1. Context

The original direct-copy model (ADR-0002) forced full-library processing on each run. For large collections (for example 1600 tracks), this increases USB NAND wear and creates unacceptable waiting times. Name-based duplicate checks are insufficient because different files may share names and FAT32 timestamps are unreliable.

## 2. Decision

Implement a diffing/sync engine with these rules:

1. Cryptographic identity: each audio file is identified by SHA256 hash. Names and mtimes are not used as source-of-truth for identity.
2. Checkpoint mirror on USB: `.provisioning_checkpoint` is persisted on the target USB so state is portable across hosts.
3. Global index continuity (`N+1`): planner reads existing indices (`001_`, `002_`, ...) and assigns the next available index for new files while preserving legacy playback order.

## 3. Consequences

- Positive:
  - Speed: post-bootstrap updates process only deltas.
  - Integrity: hash validation detects corruption/bit-rot.
  - Scalability: supports large catalogs without constant full rewrites.
- Negative:
  - Initial scan overhead to compute source hashes.
  - If checkpoint is missing/corrupted, hash-based reconstruction is required.
